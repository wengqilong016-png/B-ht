## Problem Statement

Bahati operates a network of slot machines across merchant locations in Tanzania. Each
day, field agents (drivers) visit assigned machines, read the counter, collect cash,
photograph evidence, and submit a transaction. The company (Bahati) earns revenue from
the difference between machine counter increments and driver commissions. Accuracy,
trust, and speed are the core constraints.

The current app (React 19 + Capacitor + Supabase) handles the core loop well — offline
queue, GPS capture, photo evidence, server-authoritative finance calculation. But it was
built under pressure and the codebase shows it: 12 blocking/high bugs fixed in a
2-day sprint (April 2026), open race conditions, and a QuickCollect feature shipped
without full test coverage.

The product works. The question is: what should be built next, and in what order?

## Demand Evidence

**Real, observable demand signals:**
- The app is in production use by Tanzanian field agents. Drivers use it daily for
  collections (GPS coordinates are captured, photos uploaded, transactions submitted
  to Supabase in real time).
- The codebase shows active, production-grade pressure: Sentry integration, offline
  queue with dead-letter visibility, realtime invalidation, a two-role architecture
  (driver/admin) with RLS isolation. This is not a demo.
- The QuickCollect feature (merged into driver shell) targets a real field pain: the
  full 4-step wizard (select → capture → amounts → confirm) is too slow for agents
  running 10+ machines per day. "2-3 taps per machine" is a specific UX demand signal.
- Tanzania's gaming industry collected TZS 260.21 billion in 2024/25 taxes — a
  97% surge in four years. Bahati operates in a rapidly growing, regulator-scrutinized
  market where collection accuracy directly affects tax compliance.

**Demand gaps (what the current system lacks):**
- No agent performance dashboard: drivers can't see their own payout history or route
  completion rate. Admin sees everything; agents are blind.
- No route optimization: agents pick machines manually. Efficient ordering is left to
  tribal knowledge.
- No anomaly alerting to drivers: SmartInsights exists in admin, but drivers don't see
  real-time flags about machines that look suspicious before they arrive.
- No structured dispute flow: if a driver and admin disagree on a reading, there is no
  in-app escalation path.

## Status Quo

**How field agents work today (without the app helping):**
- WhatsApp groups for shift coordination
- Physical logbooks as backup when offline
- Calling the office to report machine anomalies
- Manual cash counting and reconciliation at end of route
- No visibility into whether their submitted transactions cleared until the next morning

**Current workarounds inside the app:**
- Dead-letter queue exists but only admins can see it — agents who hit a sync failure
  have no self-service recovery path
- Offline data survives in IndexedDB but the UI gives minimal feedback about queue depth
  and expected sync time
- QuickCollect was added to reduce taps, but still requires the driver to know which
  machine to select (no smart ordering)

## Target User & Narrowest Wedge

**Primary: The field agent (driver)**
- Title: Driver / Field Agent
- What gets them promoted: high collection rate, low shortage reports, fast route
  completion
- What gets them fired: shortage (netPayable shortfall), photo evidence missing,
  late submission, GPS spoofing
- What keeps them up at night: cash in pocket that didn't sync before the app crashed

**Secondary: The settlement admin**
- Title: Admin / Finance Manager
- Daily job: DailySettlement reconciliation — matching actualCash against
  expectedTotal, catching shortages, approving exceptions
- What they lose sleep over: a driver's transaction that never synced, leaving a
  gap in the settlement ledger

**Narrowest wedge (what someone would pay for this week):**

> "A driver can open the app in the morning, see their route sorted by GPS proximity,
> tap each machine, enter the reading, and submit — without ever touching a navigation
> menu or selecting from a list."

This is achievable in ~2 days of work and eliminates the #1 driver complaint
(time-per-stop). It's not strategic, it's operational, but operational wins in field
ops compound fast: faster drivers = more collections per day = more revenue for Bahati.

## Constraints

- **Stack is fixed:** React 19 + TypeScript + Vite + Capacitor + Supabase + Tailwind 4
- **Offline-first is non-negotiable:** Tanzania network coverage is inconsistent.
  Any feature that breaks without connectivity fails the product.
- **Server is authoritative on finance:** `submit_collection_v2` / `calculate_finance_v2`
  are the source of truth. Local previews are estimates only. Designs must not create
  user expectations that conflict with server recalculation.
- **RLS isolation must be preserved:** Drivers see only their assigned locations.
  Any route-optimization feature must work within filtered data.
- **Regulatory context:** Tanzania GBT actively audits operators. The photo + GPS
  evidence chain must be tamper-evident and complete. Any shortcut here is a legal risk.
- **Two languages:** Swahili (sw) and Chinese (zh). All new strings go through i18n.
- **QuickCollect tests are blocked** by a pre-existing React 19 `act()` issue —
  new features that touch QuickCollect need to work around this until it's resolved.

## Premises

1. **The core submission flow is correct.** Server-authoritative finance + offline queue
   + evidence storage is the right architecture. We are not redesigning this.
   — Agree

2. **Driver experience is the fastest unlock.** Admin has SmartInsights, settlement,
   payroll. The driver UI was built for correctness, not speed. The next 20% of product
   investment should weight 70% toward driver UX.
   — Agree

3. **Route optimization via GPS proximity is the highest-value driver feature.**
   Sorting machines by current GPS distance is a small build (client-side, no backend
   required) with a high daily multiplier effect.
   — Agree

4. **Agent self-service visibility into sync state is a trust problem, not a technical
   one.** Drivers who can't see their queue depth or sync status will call the office
   or assume their submission was lost. This erodes trust. The offline queue already
   tracks this data; surfacing it is a UI problem only.
   — Agree

5. **Dispute resolution needs a structured path before the product scales.**
   Today, disputes go through WhatsApp. This does not scale past ~5 drivers without
   someone dropping a message. A lightweight dispute/comment thread on a transaction
   is table stakes.
   — Agree

## Landscape Awareness

**Layer 1 — conventional wisdom:**
- RouteGo (VMA Systems) is the established US/EU solution for slot machine route
  collection management. It covers machine inventory, collections processing, service
  management, prize inventory, and ATM refill. Built for Western markets, desktop-first.
- BeatRoute and WeCollect are Africa-focused field agent platforms (20,000+ users)
  with offline capability and geo-tagging. Not gaming-specific.

**Layer 2 — current discourse:**
- Tanzania gaming sector is rapidly formalizing: 14,124 licenses planned for 2025/26,
  active enforcement against ~1,846 illegal machines, foreign ownership ban enacted
  July 2025. Compliance and audit trails are becoming legally mandatory, not optional.
- FATF greylist removal (June 2025) opens Tanzania to more formal investment.

**Layer 3 — where conventional wisdom is wrong for Bahati:**

EUREKA: Every existing route collection tool (RouteGo, NCEMS) assumes a desktop admin
operator and a dumb field agent with a basic handheld. Bahati's field agents are
smartphone-native, working in a highly compliant market where photo + GPS evidence has
legal weight. The mobile-first, evidence-first design is not a tradeoff — it's the
regulatory moat. Building a better desktop admin is the wrong bet. Building a smarter
driver app is the asymmetric one.

## Approaches Considered

### Approach A: Driver Route Intelligence (Minimal Viable)
  Summary: Add GPS-sorted machine list to QuickCollect. Driver sees machines ordered
           by proximity to current location. One-tap to expand and submit.
  Effort:  S (1-2 days)
  Risk:    Low
  Pros:
    - Works within existing QuickCollect component (no new screens)
    - No backend changes — sort is client-side using already-captured GPS
    - Measurable: reduces time-per-route, testable by drivers in the field
  Cons:
    - Doesn't address sync visibility or dispute resolution
    - GPS sort only useful when driver is already in the field (not pre-route planning)
  Reuses:  useGpsCapture, filteredLocations, QuickCollect, haversine distance

### Approach B: Driver Transparency Layer (Ideal Architecture)
  Summary: Build a dedicated driver dashboard showing: today's route with GPS ordering,
           sync queue depth and last-synced timestamp, per-machine status (submitted /
           pending / failed), and a dispute comment thread on each transaction.
  Effort:  L (1-2 weeks)
  Risk:    Medium
  Pros:
    - Closes all four demand gaps at once (route, sync visibility, status, dispute)
    - Driver is no longer blind — mirrors what admins already have
    - Dispute thread uses existing transaction data model (append comment field to tx)
  Cons:
    - Larger scope: new pages, new Supabase columns, new RLS policies
    - Testing surface increases significantly
    - Needs i18n strings for all new UI
  Reuses:  offlineQueue state, transactionRepository, DriverStatusPanel skeleton

### Approach C: Compliance Evidence Hardening (Creative/Lateral)
  Summary: Instead of improving UX, harden the evidence chain for regulatory readiness.
           Add tamper-evident photo hashing (SHA-256 stored server-side), GPS
           accuracy thresholds (reject submission if accuracy > 50m), and a
           read-only audit export for GBT inspections.
  Effort:  M (4-7 days)
  Risk:    Medium
  Pros:
    - Direct response to Tanzania's tightening regulatory environment
    - Makes Bahati's audit trail legally defensible, not just operationally useful
    - Differentiates from informal operators (the "dubwi" problem)
  Cons:
    - No user-visible UX improvement for drivers
    - GBT may not require this in current form — regulatory risk is assumed, not proven
    - Adds complexity to the submission pipeline (photo hashing in the hot path)
  Reuses:  evidenceStorage, scanMeterService, GPS capture pipeline

**RECOMMENDATION:** Approach B (Driver Transparency Layer), implemented in two
sub-phases: first ship the GPS route sorting (Approach A, the wedge) within the
week, then expand to the full transparency layer. This lets drivers feel the
improvement immediately while the larger build lands. Approach C should be queued
as a Q3 bet once the regulatory requirement is confirmed by Bahati's compliance contact.

## Recommended Approach

**Phase 1 (this week): GPS Route Sort in QuickCollect**

In `driver/components/QuickCollect.tsx`, sort `assignedMachines` by haversine distance
from `gpsCoords` when coordinates are available. Add a distance badge to `MachineCard`.
Fallback: alphabetical sort when GPS is null (current behavior). No backend changes.

Implementation notes:
- **PREREQUISITE (do this first):** grep `driverFlowTelemetry.ts` for QuickCollect
  event names. If `QUICK_COLLECT_MACHINE_SELECTED` and `QUICK_COLLECT_SUBMITTED` are
  missing, add them before writing the sort code. Without these events, the 20%
  time-reduction success criterion is unmeasurable. (~30 min)
- Verify `Location` type field names: confirm the schema uses `lat`/`lng` (not
  `latitude`/`longitude` or a nested `coordinates` object) before writing haversine
- Haversine distance is ~8 lines of TypeScript, no library needed
- Sort direction: ascending (nearest machine first)
- **Reactive sort:** compute `sortedMachines` via `useMemo([assignedMachines, gpsCoords])`
  so the list re-sorts automatically when GPS warms up after mount (cold start ~2-5s)
- Distance badge: display rounded distance below the machine name — meters when
  < 1 km, kilometers when >= 1 km (e.g. "340 m" or "2.1 km")
- `gpsCoords` is already passed as a prop to `QuickCollect` — the dependency listed
  below is about whether it is reliably non-null at mount time, not whether the prop
  exists
- If `gpsCoords` becomes null mid-session (e.g. driver moves indoors), retain the
  last computed sort order rather than re-sorting alphabetically, to avoid jarring
  list reorder mid-route
- **Tests:** The React 19 `act()` test blocker prevents component tests, but write
  unit tests for the haversine utility function in `__tests__/` — it's pure TypeScript,
  no React, no DOM. Keeps test coverage for the core sort logic separate from the
  blocked component tests.

**Phase 2 (next sprint, provisional — activate after Phase 1 data is back): Driver Sync + Status Visibility + Self-Service Retry**

Surface offline queue state in the driver shell:
- Add a persistent sync status bar to `DriverStatusPanel` showing queue depth,
  last sync timestamp, and per-item status (pending / syncing / failed)
- Each failed item shows a **Retry** button that calls `flushQueue()` for that item,
  with a spinner while retrying and an error code from the dead-letter queue on failure
- This converts read-only visibility into action — drivers can unstick their own
  submissions without calling the office
- No new backend required: all data is in `offlineQueue.ts` already

**Phase 3 (next sprint after Phase 2): Machine Anomaly Pre-Alert + Dispute Thread Design**

Two parallel tracks:

**Track A — Anomaly pre-alert badge (QuickCollect):**
- Add a ⚠️ badge to `MachineCard` in QuickCollect when the machine's last transaction
  was < 50% of its 7-day average (same threshold as `SmartInsights`)
- Data is already in `filteredTransactions` via `DataContext` — no backend changes
- **RLS gate (required before shipping):** verify that `filteredTransactions` in
  `DataContext` only contains the driver's own transactions. Do not derive health
  scores from data the driver's RLS policy should not expose. Grep
  `repositories/transactionRepository.ts` for the `filteredLocations` scope.

**Track B — Dispute thread design:**
- Design is gated on Open Question #3 (admin-initiated vs. driver-initiated scope)
- Do not spec the data model until that question is answered

## Open Questions

1. **QuickCollect React 19 `act()` blocker** — is this tracked? Unblocking tests
   for QuickCollect should happen before Phase 2 lands to avoid test debt compounding.

2. **GPS accuracy threshold** — what accuracy (meters) should we require before
   trusting a GPS sort? A driver inside a concrete building may get 200m accuracy.
   Degrading gracefully (sort when accurate, alphabetize when not) is safer than
   blocking submission.

3. **Dispute thread scope** — do we need admin-initiated disputes (shortage flag) or
   driver-initiated only (contesting a reading)? Clarify before designing the
   comment model.

4. **GBT compliance requirements** — is Approach C (evidence hardening) currently
   required by the Gaming Board, or anticipated? This should be confirmed before
   investing in it.

5. **Driver count** — how many active drivers does Bahati run today? This affects
   whether the sync status bar needs to handle 5 queued items or 50.

## Success Criteria

**Phase 1:**
- Median time-per-machine-submission in QuickCollect decreases by at least 20%
  (measured via `driverFlowTelemetry` events already in place)
- No regression in offline queue reliability (monitored via Sentry)

**Phase 2:**
- Zero "did my submission go through?" WhatsApp messages to admins on a shift
  where the driver used the new sync status panel
- Driver-reported sync confidence (subjective, survey) increases

**Phase 3:**
- All shortage disputes handled in-app within 5 business days of transaction creation
- Zero WhatsApp-based dispute threads for any transaction that has an in-app comment
