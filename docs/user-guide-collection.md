## 1. 两种收款模式概览

BHT 为司机提供两种收款入口，选择哪种取决于你的熟练程度：

| 特性 | HARAKA 快速收 | COLLECT 向导收 |
|------|-------------|---------------|
| 适用人群 | 老手，熟悉流程 | 新手，需要引导 |
| 操作步骤 | 2-4 次点击 | 4 步向导 |
| 批量收 | ✓ 支持（GPS 排序列表，逐台收） | ✗ 单机草稿模式 |
| 照片 | 可选（有拍照按钮） | 强要求（无照片会弹窗提示） |
| 提交后 | 2.5 秒自动折叠卡片 | 独立成功页面（绿色对勾动画） |
| 界面入口 | 首页「HARAKA」卡片 | 首页「COLLECT」按钮 |

两种模式的底层提交逻辑完全相同，都会经过同一个服务器函数 `submit_collection_v2`，
区别仅在于用户交互流程。

## 2. HARAKA 快速收模式

### 2.1 界面入口

进入司机端首页后，你会看到一个 **HARAKA** 区域的机器列表。
这个列表会根据你当前的 GPS 位置自动排序 —— 离你最近的机器排在最上面。

每台机器显示为一个卡片，卡片上显示：
- 机器名称和编号
- 机器状态标签（正常/维护/故障/停用）
- 距你的距离（如 "1.2 km"）
- 上次读数 (lastScore)
- 如果超过 7 天未收款，会显示黄色警告标识

### 2.2 操作步骤

**第 1 步：展开机器卡片**

点击你要收款的机器卡片，卡片向下展开，露出操作区域。

展开后你会看到：
```
┌──────────────────────────────────────────┐
│ 机器名称 · #MACHINE-001                  │
│ 上次读数: 12,000   状态: 正常            │
├──────────────────────────────────────────┤
│ [______]  输入当前计数器读数（纯数字）      │
│                                          │
│ 📷 拍照按钮（相机图标）                    │
│                                          │
│ 财务预览区域（输入分数后自动计算）          │
│   diff:   250 coin × 200 = 50,000 TZS    │
│   佣金:   7,500 TZS                      │
│   机主留存: 7,500 TZS                    │
│   实得:   42,500 TZS                     │
│                                          │
│ 💰 零钱兑换: [____]                      │
│ 🎁 小费:     [____]                      │
│ 🏠 机主留存: [____]  留存/现场支付 切换     │
│                                          │
│ [提交收款] 按钮（橙色大按钮）              │
└──────────────────────────────────────────┘
```

**第 2 步：输入当前读数**

在输入框中输入机器计数器的当前数字（纯整数，不含逗号或空格）。
例如机器计数器显示 12250，就输入 `12250`。

输入后，财务预览区域会**自动实时计算**并显示：
- 本次差值 (diff)
- 收入金额 (revenue)
- 佣金 (commission)
- 实得金额 (netPayable)

**第 3 步：拍照（推荐）**

点击相机按钮拍摄机器计数器照片。照片会以 base64 格式暂存。
拍照后按钮变为绿色，显示「已拍照」标识。

> ⚠️ 虽然不是强制要求，但**强烈建议拍照**。照片是收款凭证，
> 在争议时有据可查。详见 [第 5 节：照片要求](#5-照片要求)。

**第 4 步：调整零钱/小费/留存（可选）**

如果本次收款涉及零钱兑换、给小费、或机主留存分红：
- **零钱兑换 (coinExchange)**：司机用现金兑换机器的硬币，输入兑换金额 (TZS)
- **小费 (tip)**：给机主的小费金额 (TZS)
- **机主留存 (ownerRetention)**：机主本次留存的分红金额。开关控制「留存」或「现场支付」
  - 留存模式 (isOwnerRetaining=true)：金额累加到机器的 dividendBalance
  - 现场支付模式 (isOwnerRetaining=false)：金额当场支付给机主

**第 5 步：提交**

点击橙色 **「提交收款」** 按钮。

系统会做以下校验：
- 分数不能为空
- 分数不能低于上次记录（如果低于，说明机器可能被重置过，需要先提交重置申请）
- 在线时提交到服务器；离线时自动进入离线队列

**第 6 步：查看结果**

提交成功后：
- 卡片顶部显示绿色回执单（receipt），包含：
  - ✓ 提交状态（云端已保存 / 待同步）
  - 交易 ID
  - 本次读数、收入、实得金额
- 2.5 秒后卡片自动折叠，表单清空，可以继续收下一台

提交失败时：
- 卡片顶部显示红色错误信息
- 提交锁解除，可以修改后重试

### 2.3 HARAKA 快速收的关键特点

- **批量收**：多台机器可同时保持展开状态，每台独立管理分数、照片、提交状态
- **GPS 排序**：机器列表自动按距离排序，最近的优先
- **自动折叠**：成功后自动收起，减少屏幕占用
- **数据刷新**：提交后自动刷新位置、交易、司机数据缓存

## 3. COLLECT 向导模式

### 3.1 界面入口

点击首页的 **「COLLECT」** 按钮进入向导流程。

向导共 4 步，顶部有步骤指示条（WizardStepBar）：

```
 ● 选机器  ──  ○ 读数拍照  ──  ○ 调整金额  ──  ○ 确认提交
```

### 3.2 Step 1: 选择机器 (selection)

界面显示你被分配的所有机器列表（MachineSelector 组件）。

每台机器显示：
- 机器名称和编号
- 所在区域
- 状态标签
- 上次读数

点击一台机器进入下一步。底部导航栏可切换回 HARAKA 快速收。

### 3.3 Step 2: 读数与拍照 (capture)

**ReadingCapture 组件** —— 这是向导模式的核心输入步骤。

界面上半部分：
```
┌──────────────────────────────────────┐
│ 机器: Gerezani Shop · #M-042         │
│ 上次读数: 12,000                     │
├──────────────────────────────────────┤
│                                      │

│                                      │
├──────────────────────────────────────┤
│ [返回选机器]           [下一步 →]     │
└──────────────────────────────────────┘
```

- 输入当前计数器读数
- 拍摄机器计数器照片

**SubmitReview 组件** —— 最后一步，确认所有信息并提交。

界面布局（从上到下）：

## 4. 财务预览怎么看

### 4.1 计算公式

每次收款提交的财务计算遵循以下公式（由服务器权威计算）：

```
diff              = MAX(0, currentScore - lastScore)
                    （当前读数 - 上次读数，最小为 0）

revenue           = diff × 硬币价值
                    （硬币价值从 constants 表动态读取，约 200 TZS/coin）

commission        = FLOOR(revenue × commissionRate)
                    （佣金 = 收入 × 佣金率，向下取整）

finalRetention    = 机主留存金额（如果手动填写则用该值，否则默认等于 commission）

available         = MAX(0, revenue - finalRetention - |expenses| - |tip|)
                    （可用金额 = 收入 - 留存 - 费用绝对值 - 小费绝对值）

startupDebtDeduction = MIN(填写的抵扣金额, 剩余创业债, available)
                    （创业债抵扣，受三重限制）

netPayable        = MAX(0, available + startupDebtDeduction)
                    （实得 = 可用金额 + 创业债抵扣，最小为 0）
```

### 4.2 财务预览界面解读

无论是在 HARAKA 模式还是向导模式，财务预览都显示以下字段：

| 字段 | 英文 | 含义 | 示例 |
|------|------|------|------|
| **diff** | Diff | 本次计数器增长（coin 数） | 250 |
| **revenue** | Revenue | 本次总收入 (TZS) | 50,000 |
| **commission** | Commission | 系统佣金 (TZS) | 7,500 |
| **ownerRetention** | Owner Retention | 机主留存分红 (TZS) | 7,500 |
| **expenses** | Expenses | 费用（收款类型始终为 0） | 0 |
| **tip** | Tip | 小费 (TZS) | 0 |
| **startupDebtDeduction** | Debt Deduction | 创业债抵扣 (TZS) | 0 |
| **netPayable** | Net Payable | 司机实得 (TZS) | 42,500 |
| **remainingCoins** | Remaining Coins | 剩余硬币库存 | — |

### 4.3 红色显示和警告

- **实得 (netPayable) 为负数时**：显示为红色，表示本次收不抵支
- **零钱库存不足 (isCoinStockNegative)**：提示司机硬币不够兑换
- **diff = 0**：分数没涨，收入为 0（系统仍会记录，标记为零收入提交）

### 4.4 计算时机

- **HARAKA 模式**：每次修改分数后自动重新计算（本地 `calculateCollectionFinanceLocal`）
- **向导模式**：在 Step 3 (amounts) 中显示，修改数值后实时更新
- **提交时**：服务器用 `submit_collection_v2` 重新权威计算，覆盖客户端预览值

## 5. 照片要求

### 5.1 照片的重要性

照片是收款凭证（evidence），在提交到服务器前必须完成上传。

> ⚠️ **照片分类为 `kind='evidence'` 的错误**：如果照片缺失或上传失败，在线提交会**直接拒绝**，
> 不会降级到离线队列（因为离线也无法解决照片问题）。

### 5.2 照片要求

- **内容**：拍摄机器计数器读数，清晰可见数字
- **格式**：支持 JPEG/PNG，自动压缩到合理大小
- **大小**：上传前自动 resize（宽 1280px，质量 0.7）
- **来源**：手机相机拍摄，或从相册选择

### 5.3 照片流程

```
拍照 (base64 Data URL)
  │
  ▼
前端暂存 (photoData state)
  │
  ▼
提交时 → persistEvidencePhotoUrl()
  │
  ├─ 上传到 Supabase Storage
  │   路径: evidence/{category}/{timestamp}_{random}.jpg
  │
  └─ 返回公开 URL → 传递给 submit_collection_v2 RPC
       │
       ▼
    存入 transactions.photoUrl 字段
```

### 5.4 按钮变红提示

- **HARAKA 模式**：拍照按钮在没有照片时为默认颜色，拍照后变为绿色。如果照片因页面刷新丢失，会显示警告文字
- **向导模式**：
  - 没有照片时，Step 4 确认页显示**黄色警告条**：「照片在刷新后丢失，请返回上一步重新拍照」
  - 点击提交按钮时，如果没有照片会**弹出确认框**
  - 如果提交列表中有「需要拍照」(submissionBlockers)，提交按钮会变灰禁用

### 5.5 EXIF GPS 提取

如果用户没有授予 GPS 权限但有照片，系统会尝试从照片的 EXIF 数据中提取 GPS 坐标：
- 成功提取 → 使用 EXIF GPS 坐标提交
- 提取失败 → 回退到网点坐标（需用户确认）

## 6. 离线收款怎么处理

### 6.1 离线检测

系统通过 `isOnline` 状态检测网络连接。以下情况自动进入离线模式：
- 设备无网络连接
- 服务器不可达（超时/网络错误）
- RPC 调用超时（30 秒硬超时）

### 6.2 离线提交流程

```
用户点击「提交」
  │
  ▼
前端校验通过
  │
  ▼
orchestrateCollectionSubmission()
  │
  ├─ isOnline = true?
  │    ├─ 调用 submitCollectionV2() → 成功 → 返回 {source:'server'}
  │    └─ 调用失败（非 evidence 类错误）
  │         └─ fallbackToOffline() → 离线队列
  │
  └─ isOnline = false
       └─ 直接 fallbackToOffline() → 离线队列
```

### 6.3 离线队列存储

离线交易存储在浏览器 IndexedDB 中：
- 数据库名：`bahati_offline_db`
- 存储表：`pending_transactions`
- 每条记录包含完整交易数据 + 原始提交参数 (rawInput)

### 6.4 离线队列同步

恢复在线后，系统自动触发同步：
1. 提交成功后检查队列健康状态（`getQueueHealthSummary`）
2. 如果有待处理条目 → 自动调用 `flushQueue()`
3. 按 FIFO 顺序逐条重放到服务器
4. 服务器用 `rawInput` 重新权威计算财务值，覆盖客户端本地计算值

### 6.5 离线状态下的用户体验

- **HARAKA 模式**：提交成功后 receipt 显示 `status: 'offline'`，提示「已加入待同步队列」
- **向导模式**：成功页面显示「待同步」而非「云端已保存」
- 导航栏可能出现离线图标 (WifiOff)
- 部分依赖在线的功能（如编辑个人资料）会被禁用

### 6.6 重试机制

- 每条离线记录有 `retryCount` 计数器
- 失败后使用指数退避（exponential backoff）延迟重试
- `nextRetryAt` 字段记录下次可重试的最早时间
- 永久性错误（如认证失败）会标记为 `lastErrorCategory: 'permanent'`

## 7. MY STATUS 怎么看

### 7.1 入口

点击底部导航栏的 **「MY STATUS」**（或驾驶员头像图标）进入个人状态页面。

该页面由 `DriverStatusPanel` 组件渲染，包含以下几个区域：

### 7.2 个人信息卡片（顶部）

```
┌──────────────────────────────────────┐
│  [头像首字母]  司机姓名               │
│               用户名                  │
│               电话号码                │
│                         [在线] 标签   │
├──────────────────────────────────────┤
│ 🕐 最后活跃: May 10, 14:30           │
└──────────────────────────────────────┘
```

- **头像**：显示司机姓名首字母（大写），圆角方形
- **姓名/用户名**：司机的显示名和登录用户名
- **电话**：司机登记的电话号码
- **在线状态标签**：

  - 黄色 `AWAY`：暂时离开
  - 红色 `OFFLINE`：离线
- **最后活跃时间**：最近一次操作的时间戳
- **背景图**：可上传个人背景照片

### 7.3 编辑个人资料

可编辑字段：
- **电话号码**：用于联系司机
- **背景照片**：个人资料页的背景图片

## 8. 常见问题

### Q: 分数输入后为什么显示「当前读数低于上次记录」？

A: 你输入的分数比机器记录的 `lastScore` 还低。可能原因：
- 机器计数器被重置过了 → 需要先提交**重置申请** (Reset Request)
- 输入错误 → 核对机器计数器后重新输入

在 HARAKA 模式下直接拒绝提交；向导模式下提交按钮被禁用。

### Q: 提交时提示「照片缺失」怎么办？

A: 返回 Step 2 拍摄照片，或者在弹窗中选择「继续提交」（不推荐）。
注意：照片可以作为收款凭证，有争议时非常重要。

### Q: 离线提交的数据什么时候同步？

A: 恢复网络后自动同步。你也可以在提交成功后手动触发同步。
离线数据按 FIFO 顺序逐条重放，每条数据会经过服务器重新计算。

### Q: 零钱兑换 (coinExchange) 和小费 (tip) 的区别？

A:
- **零钱兑换**：司机用现金从机器换硬币（coin → TZS），影响实得金额
- **小费**：给机主的额外小费（TZS），从收入中扣除

### Q: 机主留存 (ownerRetention) 和现场支付有什么区别？

A: 切换 `isOwnerRetaining` 开关控制：
- **留存 (true)**：分红金额累积到机器的 `dividendBalance`，机主稍后统一提现
- **现场支付 (false)**：分红金额当场支付给机主，不进入 dividendBalance

### Q: 什么是「创业债抵扣 (startupDebtDeduction)」？

A: 机器有初始的启动债务（机器采购/安装成本），司机每次收款时可选择抵扣一部分。
抵扣金额受三重限制：不能超过你填写的金额、不能超过剩余债务、不能超过可用金额。
抵扣会减少债务余额，同时增加你的实得金额。

### Q: HARAKA 和 COLLECT 的数据互通吗？

A: 完全互通。两者共享相同的 `locations`（机器数据）、`transactions`（交易数据）、
`drivers`（司机数据）。在 HARAKA 提交后，COLLECT 向导也能看到最新的 lastScore；
反之亦然。

### Q: 每天能对同一台机器收几次款？

A: 技术上可以多次收款，但当天第二次提交时会弹出确认框：「今天已对此机器提交过一次收款记录。是否确认再次提交？」
这是为了防止误操作。

### Q: 硬币价值 (coin value) 是多少？

A: 硬币价值由系统管理员在 `constants` 表中配置，当前约 **200 TZS/coin**。
该值可通过管理面板动态调整，所有计算自动使用最新值。
