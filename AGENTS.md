# AGENTS.md

## 项目目标
这是一个需要真实可用、可上线运行的项目。
修复问题时不能只“消报错”，目标是恢复真实功能，并通过可复现的验证证明修复有效。

## 核心原则
- 先理解再修改：先读相关代码与调用链，再动手。
- 先根因后修复：优先定位根因，不做拍脑袋改动。
- 最小必要改动：只改与当前问题直接相关的代码。
- 不重复失败路径：同一思路失败后，必须更换策略。
- 不以测试为代价破坏业务：不能为了过测试注释/阉割功能。
- 保持行为一致：接口契约和业务目标不应被意外改变。

## 收到 Bug 后的标准流程
1. 明确问题：记录「复现步骤 / 预期行为 / 实际行为」。
2. 缩小范围：阅读相关模块、入口组件、数据流和副作用逻辑。
3. 根因判断：列出可能原因并按概率排序。
4. 实施修复：先做概率最高且最小的方案。
5. 执行验证：先快速验证，再按需要做完整验证。
6. 失败迭代：根据新报错继续收敛，避免重复同类方案。
7. 结束说明：给出根因、改动、验证结果、剩余风险。

## 明确禁止
- 不要跳过验证就宣称修复完成。
- 不要通过注释/删除功能来伪装修复。
- 不要删除报错代码而不解决根因。
- 不要无依据地升级/降级依赖版本。
- 不要一次改动大量无关文件。
- 不要编造命令结果或测试结果。

## 修改范围约束
- 仅修改与当前问题直接相关的文件。
- 遵循现有代码风格、命名习惯和目录组织。
- 如必须扩大范围，先在回复中说明原因与影响面。

## 本仓库验证规范

### 0) 依赖安装（按需）
- 仅在以下情况执行：首次拉仓库、`package-lock.json` 变化、`node_modules` 缺失。
- 命令：`npm ci`

### 1) 快速验证（默认）
- 目标：尽快判断修复是否有效，适用于迭代中。
- 建议顺序：
  - `npm run lint`
  - `npm run test:ci`（或与改动最相关的测试）
  - `npm run build`

### 2) 完整验证（提交前/发布前）
- 目标：降低回归风险，适用于合并前、发版前。
- 建议顺序：
  - `npm run lint`
  - `npm run test:ci`
  - `npm run build`
  - 如涉及安卓打包/插件：`npm run cap:build:android` 或 CI 的 Android workflow
  - 如涉及关键页面流程：执行对应手工路径验证（写明路径与结果）

### 3) 命令缺失时处理
- 若命令不存在，先检查 `package.json` 与 `README.md`，选择等价命令。
- 若环境限制导致命令无法运行，必须明确说明「未运行项 + 原因 + 风险」。

## 输出要求（每次任务结束必须包含）
- 根因是什么（或当前最可能根因）。
- 修改了哪些文件。
- 每个修改的目的是什么。
- 运行了哪些验证命令。
- 哪些验证通过、哪些未通过（未通过要给原因）。
- 当前剩余风险或阻塞点。

## 质量红线
- 不得声称“已验证”但实际未执行。
- 不得省略关键失败信息。
- 若对结果没有把握，明确标注不确定性并给出下一步验证方案。
